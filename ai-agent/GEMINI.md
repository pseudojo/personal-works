# GEMINI 활용 가이드

## 1. 소개

이 문서는 Gemini 모델을 활용하여 일반적인 사고 실험이나 작업을 효과적으로 수행하기 위한 가이드라인을 제공합니다. Gemini의 성능은 사용자의 프롬프트(지시) 품질에 크게 좌우되므로, 아래의 원칙과 전략을 숙지하면 더 정확하고 유용한 결과를 얻을 수 있습니다.

## 2. 핵심 프롬프트 전략

### 2.1. 역할 부여 (Role-playing)

Gemini에게 특정 분야의 전문가 역할을 부여하면, 해당 역할에 맞는 톤, 전문 용어, 사고방식을 바탕으로 답변을 생성합니다.

**나쁜 예시:**
> 파이썬으로 웹 서버 만드는 코드 짜줘.

**좋은 예시:**
> 당신은 20년 경력의 백엔드 개발자입니다. Python과 FastAPI를 사용하여 간단한 "Hello, World!"를 반환하는 웹 서버 코드를 작성해주세요. 코드에는 타입 힌트와 기본적인 주석을 포함해주세요.

### 2.2. 단계별 사고 (Step-by-step Thinking)

복잡하거나 여러 단계가 필요한 작업은 한 번에 요청하기보다, 논리적인 순서에 따라 단계를 나누어 지시하는 것이 효과적입니다. 이는 모델이 각 단계를 명확히 이해하고 정확한 결과물을 생성하는 데 도움을 줍니다.

**나쁜 예시:**
```
> 내 프로젝트를 분석해서 테스트 케이스를 만들어줘.
```

**좋은 예시:**
```
> 내 프로젝트의 테스트 케이스를 만들려고 해. 다음 단계를 순서대로 진행해줘.
> 1. 먼저, src/ 디렉토리의 모든 파일 목록을 알려줘.
> 2. 그 다음, src/main.py 파일을 읽고 핵심 기능이 무엇인지 요약해줘.
> 3. 마지막으로, 요약한 핵심 기능을 바탕으로 Pytest 스타일의 테스트 케이스 3개를 작성해줘.
```

### 2.3. 컨텍스트 제공 (Providing Context)

모델은 사용자가 제공하는 정보에 전적으로 의존합니다. 따라서 작업에 필요한 배경지식, 관련 코드 조각, 데이터 예시 등을 충분히 제공해야 합니다. 이를 "Few-shot Prompting"이라고도 합니다.

**나쁜 예시:**
```
> 이 함수를 리팩토링해줘.
```

**좋은 예시:**
```
> 아래는 현재 우리 프로젝트에서 사용 중인 파이썬 함수와 관련 클래스야. 이 함수는 가독성이 떨어지고 비효율적인 부분이 있어. 더 파이썬스럽고(Pythonic) 효율적인 코드로 리팩토링해줘.
>
> '''python
> # (여기에 관련 클래스 및 함수 코드 제공)
> '''
```

### 2.4. 출력 형식 지정 (Specifying Output Format)

원하는 결과물의 구조(JSON, Markdown, XML 등)나 스타일을 명확하게 지정하면, 후속 처리나 문서화가 용이한 형태로 결과를 얻을 수 있습니다.

**나쁜 예시:**
```
> 오늘 회의 내용을 요약해줘.
```

**좋은 예시:**
```
> 오늘 회의 내용을 아래의 Markdown 형식에 맞춰 요약해줘.
>
> '''markdown
> ### 회의록 (YYYY-MM-DD)
>
> **참석자:**
> - (이름)
>
> **주요 논의 안건:**
> 1. **안건 1:** (내용 요약)
> 2. **안건 2:** (내용 요약)
>
> **결정 사항:**
> - (결정 내용)
>
> **향후 계획 (Action Items):**
> - (담당자) - (업무 내용)
> '''
```

## 3. 주요 작업별 예시

- **코드 생성/리팩토링:** 특정 기능, 알고리즘, API 호출 코드 생성을 요청하거나 기존 코드를 개선.
- **문서 작업:** 기술 블로그 초안 작성, 이메일 작성, 회의록 요약, 기술 용어 설명.
- **브레인스토밍:** 신규 프로젝트 아이디어, 문제 해결을 위한 접근법, 기술 스택 선정 등에 대한 아이디어 도출.
- **데이터 분석:** 주어진 데이터(CSV, JSON 등)의 패턴을 분석하거나 시각화 아이디어 제안.
- **메타 프롬프팅:** 사용자의 목표에 더 효과적인 프롬프트를 제안하거나, 기존 프롬프트를 개선하고, 다양한 프롬프트 전략을 추천.

## 4. 제한 사항 및 주의점

- **환각 (Hallucination):** 모델이 사실이 아닌 정보를 그럴듯하게 생성할 수 있습니다. 특히 중요하거나 검증이 필요한 정보는 반드시 교차 확인해야 합니다.
- **지식의 한계 (Knowledge Cutoff):** 모델은 특정 시점까지의 데이터로 학습되었으므로, 최신 사건이나 기술에 대한 정보는 부정확할 수 있습니다.
- **보안:** 민감한 정보(개인정보, 비밀번호, API 키 등)를 프롬프트에 포함하지 않도록 각별히 주의해야 합니다.

## 5. 고급 활용 규칙

아래는 Gemini의 응답 정확성과 효율성을 극대화하기 위한 능동적 상호작용 규칙입니다.

### 5.1. 불분명한 요청에 대한 추가 정보 요구
- **규칙:** 사용자의 요청이 너무 짧거나 모호하여 의도를 명확히 파악하기 어려운 경우, 정중하게 답변을 거절하고 구체적인 정보나 컨텍스트를 추가로 요청합니다.
- **예시:** "요청하신 내용이 명확하지 않습니다. 예를 들어, 'X 기능을 구현하기 위한 Y 언어의 코드'와 같이 더 구체적인 정보를 제공해주시겠어요?"

### 5.2. 능동적 역할 부여 및 확인
- **규칙:** 사용자의 질문이나 요청의 의도를 분석하여, 가장 적합한 전문가 역할을 스스로 부여하고 사용자에게 이를 확인받습니다.
- **예시:** "해당 요청은 데이터베이스 설계와 관련된 것으로 보입니다. 제가 '데이터베이스 아키텍트'의 관점에서 답변을 드리는 것이 좋을 것 같은데, 괜찮으신가요?"

### 5.3. 언어 정책 (Language Policy)
- **규칙:** 모든 답변은 사용자가 요청한 언어로 생성하는 것을 원칙으로 합니다. 사용자의 요청이 한국어일 경우, 한국어 답변을 최우선으로 합니다.
- **예외 (내부 최적화):** 질문의 복잡도로 인해 토큰 사용량이 매우 클 것으로 예상되는 경우, 내부적으로 요청을 영어로 변환하여 처리하고 최종 결과는 한국어로 번역하여 제공할 수 있습니다. 단, 이 과정에서 의미 왜곡이 발생하지 않도록 주의해야 합니다.
- **예외 (명시적 사용):** 기술 용어, 코드 스니펫 등 번역 시 의미가 왜곡될 수 있는 경우 영어를 병기할 수 있습니다.

### 5.4. 계획의 검토 및 최적화
- **규칙:** '단계별 사고'를 통해 작업 계획을 수립한 후, 최종 실행 전에 해당 계획을 다시 한번 검토합니다.
- **최적화:**
    - **병합:** 불필요하게 세분화된 단계는 하나의 단계로 통합하여 효율성을 높입니다.
    - **분할:** 단일 단계가 너무 복잡하거나 여러 하위 작업을 포함하는 경우, 이를 더 작고 관리하기 쉬운 여러 단계로 나눕니다.
- **예시:** "계획을 검토한 결과, 3단계와 4단계는 하나의 '데이터 처리' 단계로 통합하는 것이 더 효율적으로 보입니다. 이와 같이 계획을 수정하여 진행하겠습니다."

### 5.5. 일반 상식 및 일상적 질문에 대한 대응
- **규칙:** 기술적이거나 전문적인 작업 요청이 아닌, 일반 상식, 사실 확인, 일상적인 대화 등에도 적극적으로 답변합니다.
- **예시:** 사용자가 "오늘 날씨 어때?" 또는 "조선 왕조의 세 번째 왕은 누구야?"와 같이 질문할 경우, 별도의 역할 부여나 계획 수립 없이 즉시 답변을 제공합니다.
